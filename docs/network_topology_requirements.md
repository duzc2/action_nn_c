# 神经网络复杂拓扑需求分析

## 1. 背景

当前流程以配置头文件作为网络入口，适用于规模较小、结构较稳定的网络配置。\
当网络进入“多子网、可嵌套、跨子网混合连接”的阶段后，单个配置头文件已无法清晰承载：

- 子网层级结构与复用关系
- 子网间多对多节点连接关系
- 每类子网的专属参数集合
- 训练与推理共享的完整结构语义

因此需要将“网络配置”从头文件迁移为“用户程序内的结构化参数对象”，并以该对象驱动 profiler 生成后续训练/推理/保存/加载代码。

## 2. 目标

- 不再使用统一配置头文件描述网络
- 每个用户程序自行构建网络结构体实例
- 网络结构体可描述复杂拓扑、嵌套子网与混合连接
- profiler 直接消费该结构体，生成功能拆分的 .h/.c 文件
- profiler 生成代码主要解决用户网络描述的多变性，核心算法实现在 src
- 训练与推理均使用 profiler 生成代码进行编译与执行
- 保存与加载流程与同一份结构定义保持一致
- 网络结构变化时，对外 API 头文件保持稳定
- 所有代码必须严格遵守 C99 规范

## 3. 范围

本次需求分析覆盖以下内容：

- 结构化网络描述模型的能力边界
- profiler 的输入、输出与代码生成职责
- 训练/推理/保存/加载四类流程的统一关系
- 面向复杂拓扑的校验、约束与失败处理要求

本次不包含：

- 具体 C 文件实现
- 已有代码迁移的逐文件改造步骤
- 任意未被需求明确提出的新业务概念

## 4. 核心概念定义

- 网络定义（Network Definition）：用户程序中构建的结构化对象，描述完整网络图。
- 子网（Subnetwork）：网络定义中的可独立命名单元，类型可为 RNN、CNN 等。
- 子网类型（Subnetwork Type）：子网的语义类型标签，不使用数字硬编码代表模型类型。
- 节点端口（Port）：子网输入端口与输出端口，作为连接关系的挂接点。
- 连接（Edge）：从一个子网输出端口到另一个子网输入端口的有向连接。
- 嵌套（Nested Subnetwork）：子网内部可继续包含子网，形成层级结构。
- profiler：读取网络定义，完成校验、展开、代码生成与构建产物组织。

## 5. 功能需求

### 5.1 网络结构体表达能力

- 必须支持多个子网并存。
- 每个子网必须支持以下最小属性：
  - 子网唯一标识
  - 子网类型（如 RNN、CNN）
  - 子网参数集合（类型相关属性）
  - 隐藏层数量（允许为 0）
  - 每个隐藏层的节点数量
  - 默认激活函数
  - 节点级激活函数覆盖定义（可选）
  - 输入层节点数量
  - 输出层节点数量
  - 输入节点定义（按节点索引可定位）
  - 输出节点定义（按节点索引可定位）
- 必须支持子网嵌套，且嵌套深度不限制为单层。
- 必须支持“一个子网输出混合接入多个目标子网输入”的多分支连接。
- 必须支持“一个目标输入由多个来源输出汇入”的多源连接，连接合并策略需被显式描述。
- 必须支持同级连接与跨层连接的统一表示。

### 5.2 用户程序职责

- 用户程序负责构建网络结构体实例，不再依赖配置头文件。
- 用户程序负责在调用 profiler 时将已构建的网络结构体作为函数参数直接传入。
- 用户程序负责在训练与推理构建阶段指定要使用的网络定义实例。
- 用户程序负责通过参数传递每个功能模块的 `.c/.h` 输出路径。
- 用户程序可为不同模块指定不同目录，实现分开存储。
- 用户程序不需要为 profiler 提供回调接口。

### 5.3 profiler 职责

- 接收网络结构体作为唯一结构输入源。
- 对外提供显式函数接口，输入参数为用户构建完成的网络结构体实例。
- profiler 调用链路不依赖用户回调函数。
- 对网络定义执行完整一致性校验，至少包含：
  - 标识唯一性
  - 端口存在性
  - 连接合法性
  - 有向无环（DAG）校验
  - 嵌套展开合法性
- 若检测到环或其他关键校验失败，profiler 必须在运行阶段立即报错并停止后续工作。
- profiler 必须在遇到第一个错误时尽快失败返回，不继续执行后续校验与生成步骤。
- 错误信息必须同时输出到终端，并通过函数参数中的错误字符串指针写回调用方。
- profiler 必须返回明确错误码，用于调用方判定失败类型。
- 错误描述必须详细、可读、可准确定位错误来源（子网、节点、连接或字段）。
- 错误描述文本必须使用纯英文。
- 建议调用方传入的错误描述缓冲区长度不小于 256 字符。
- 生成训练与推理共享的目标代码与元数据。
- 生成保存与加载所需的结构映射信息，保证参数持久化可逆。
- 基于用户传入的网络描述数据结构计算网络签名 Hash，并用于权重文件一致性校验。
- 在生成的网络权重读取 .c 文件中固化该 Hash 常量，用于运行时比对。
- 按用户参数指定的模块级路径输出 `.c/.h` 文件，支持分模块分目录存放。
- 生成代码只负责网络可变部分，算法主逻辑不得从 src 迁移到生成代码。
- 根据用户网络规则生成以下功能拆分模块的 .c 文件：
  - tokenizer
  - 网络初始化
  - 参数加载
  - 训练
  - 参数保存
  - 推理
- .h 文件不由 profiler 按配置生成，而是从源代码中的固定头文件模板复制输出。
- 复制输出的 .h 文件必须包含描述当前网络的注释内容。
- 生成/复制后的功能模块必须可独立编译与链接，支持被不同程序分别集成与部署。
- 代码生成阶段完成后，由用户在后续步骤分别编译训练环境与推理环境。

### 5.4 训练与推理职责

- 训练路径必须依赖 profiler 生成代码，而非直接读取配置头文件。
- 推理路径必须依赖同一份 profiler 生成代码，禁止绕开结构定义链路。
- 训练与推理必须共享同一网络语义与参数布局。
- 训练与推理都必须同时提供两种能力：
  - 自主循环能力（模块内部控制完整循环）
  - 单步触发能力（由外部代码控制循环，每次只触发一次迭代）
- 自主循环能力与单步触发能力同等重要，需面向不同场景并行支持。
- 保存与加载必须与生成代码中的参数索引保持一致。
- 权重保存时必须写入基于当前网络描述计算得到的 Hash。
- 每次读取权重文件时必须验证文件内 Hash 与当前加载逻辑中的 Hash 常量一致。
- Hash 不一致时必须立即报错并拒绝加载，禁止不同网络权重混用。
- 训练过程必须支持参数检查点保存，且支持从检查点加载后继续训练。
- 继续训练时必须保持网络结构定义一致，避免结构变化导致参数错位。
- 必须支持基于损失指标管理多个训练版本，用于逐步对比并选择最合适版本。

## 6. 数据模型要求（分析级）

### 6.1 结构组成

网络定义至少应包含以下层级：

- 网络级属性（名称、版本、默认精度等）
- 子网列表（支持嵌套）
- 连接列表（全局连接视图）
- 端口类型与形状描述
- 可选的执行提示（如并行组、设备偏好）

### 6.2 子网属性分类

子网属性应按“通用属性 + 类型扩展属性”组织：

- 通用属性：标识、名称、类型、输入输出端口、子网层级信息
- 通用属性必须包含隐藏层数量与每层节点数量，且明确允许 0 层隐藏层配置。
- 激活函数策略采用“默认激活函数 + 节点级覆盖”的双层机制。
- 类型扩展属性：由子网类型决定，例如 RNN 的时间步相关参数、CNN 的卷积相关参数

### 6.3 连接语义要求

- 连接必须显式记录源子网、源端口、目标子网、目标端口。
- 连接必须支持“源输出节点索引 -> 目标输入节点索引”的节点级映射。
- 节点级映射必须允许对每个输出节点单独指定目标网络与目标输入节点。
- 节点级映射必须支持“单个输出节点同时接入多个目标网络输入节点”的一对多分发。
- 连接必须支持附加映射信息（如切片、拼接、重排）以表达混合接入。
- 连接关系应可被 profiler 转换为确定性执行序列或等价执行图。

## 7. 关键约束

- 禁止回退到统一配置头文件模式。
- 结构定义必须是训练与推理共享的单一事实来源。
- 子网类型必须语义化命名，不使用二值模型开关。
- 网络定义、生成代码、参数保存格式三者必须可追溯且一致。
- 当校验失败时，profiler 必须给出可定位到子网或连接的错误信息。
- 网络拓扑必须满足有向无环；若存在环路，profiler 必须报错并终止生成流程。
- 检查点加载继续训练时，必须校验结构签名与参数布局一致性，不一致时必须拒绝加载。
- 对外 API 头文件必须固定；网络结构变化时不得改变公开函数签名与调用方式。
- .h 文件内容来源必须是固定模板复制，不允许按网络配置动态生成声明结构。
- profiler 输入模式固定为“函数参数传入网络结构体”，不引入用户回调注册模式。
- profiler 错误处理必须采用“终端输出 + 错误字符串指针写回 + 错误码返回”的统一机制。
- profiler 必须在首个错误处快速失败并停止后续流程。
- 网络权重加载必须通过 Hash 校验绑定到当前网络描述，禁止跨网络加载。
- 训练与推理接口必须同时保留自主循环入口与单步触发入口，不允许只保留其中一种。
- 所有网络实现必须放在 `src/nn/[网络类型]/` 目录下。
- 每个网络类型目录必须包含训练与推理两套 `.c/.h` 文件，不能仅保留单个 `.c/.h`。
- 每个网络类型目录必须包含 `README.md` 说明该网络目录用途与文件职责。
- 依赖方向必须满足：训练模块可依赖推理模块，推理模块不得依赖训练模块。
- 源码、模板头文件与 profiler 生成的 C 代码都必须严格符合 C99 语法与约束。
- profiler 输出路径必须由调用参数显式指定到模块级，禁止隐式写入固定目录。
- 迁移完成后工程中不得保留任何旧实现入口、旧配置路径或兼容分支代码。
- 不对历史兼容性做任何保留处理，所有调用路径必须统一到新实现。

## 8. 兼容与迁移要求

- 本项目迁移策略为“零历史兼容”：不保留旧实现支持，不保留旧调用入口。
- 迁移后工程中不得存在旧方法、旧配置入口、旧流程说明或遗留兼容代码。
- 新增与改造代码必须直接使用网络结构体 + profiler 新流程，不允许回退旧路径。

## 9. 风险与应对

- 结构复杂度提升导致构建门槛上升：通过严格校验与错误定位降低使用成本。
- 多源输入与跨层连接引入歧义：通过显式连接策略字段消除默认推断。
- 生成代码与持久化格式不一致：将参数布局映射纳入 profiler 统一输出。
- 嵌套展开后执行顺序不稳定：要求 profiler 提供确定性拓扑展开规则。

## 10. 验收标准

- 可以在单个用户程序中构建包含嵌套子网与混合连接的网络结构体。
- 可以为最小子网配置 0 层或多层隐藏层，并为每层定义节点数量。
- 可以为每个子网设置默认激活函数，并可对任意节点单独覆盖激活函数。
- 可以为每个子网独立定义输入层节点数量与输出层节点数量。
- 可以逐个输出节点配置其值流向哪个目标子网的哪个输入节点。
- 可以将单个输出节点的值同时路由到多个网络的输入节点。
- profiler 可以基于该结构体完成校验并生成训练/推理共享代码。
- 用户可通过 profiler 的函数接口直接传入网络结构体完成生成流程，且无需实现回调接口。
- 当网络包含环路时，profiler 运行会明确报错并停止，不生成后续代码产物。
- 当输入包含首个可检测错误时，profiler 会立即停止并返回错误码。
- 错误信息会同时输出到终端，并写入调用方提供的错误字符串缓冲区。
- 错误描述内容为纯英文，且可定位到具体错误来源。
- profiler 可按用户配置生成 tokenizer、初始化、加载、训练、保存、推理六类功能拆分的 .c 文件。
- profiler 可从源代码复制对应固定 .h 文件，并附加描述当前网络的注释内容。
- 用户可按模块单独指定 `.c/.h` 输出路径，并验证产物写入对应目录。
- 权重文件包含由网络描述数据结构计算得到的 Hash 字段。
- 每次读取权重文件都会执行 Hash 校验；Hash 不一致时拒绝加载并返回错误。
- 网络权重读取 .c 文件内存在与当前网络对应的 Hash 常量用于校验。
- 生成代码用于承载网络可变配置，核心算法实现位于 src 模块。
- 每个网络类型在 `src/nn/[网络类型]/` 下包含训练与推理两套实现文件，以及 `README.md`。
- 训练与推理两套实现文件均为 `.c/.h` 成对存在，不允许仅单文件对。
- 推理实现不依赖训练实现。
- 以上功能模块可分别编译并部署到不同程序中。
- 训练模块可在内部自主执行完整循环，也可由外部代码按步触发单次循环。
- 推理模块可在内部自主执行完整循环，也可由外部代码按步触发单次循环。
- 训练与推理构建过程可直接引用生成代码并完成编译。
- 网络结构变化后重新生成代码时，公开 .h API 保持不变。
- 保存与加载可在同一结构定义下互相兼容，参数位置与语义一致。
- 训练可在任意阶段保存参数检查点，并可从检查点恢复后继续训练。
- 可对多轮继续训练形成的版本进行损失对比，并选出目标版本。
- 所有源码与生成代码可在 C99 编译模式下通过编译，不依赖超出 C99 的语言特性。
- 整个链路不依赖配置头文件。
- 工程内不存在旧实现支持代码与历史兼容分支。

## 11. 建议的后续文档拆分

- 规格文档：定义结构体字段、枚举、连接策略、错误码。
- 接口文档：定义用户程序向 profiler 提供网络定义的 API 约定。
- 迁移文档：定义从配置头文件到结构体定义的迁移步骤与兼容策略。
- 测试文档：定义嵌套、混合连接、多源汇入、保存加载一致性的测试矩阵。
