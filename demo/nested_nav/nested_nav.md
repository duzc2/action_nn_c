# nested_nav demo

## 这个 demo 在模拟什么场景

这个 demo 模拟的是一种很常见的真实应用：

> 一个小车、机器人，或者游戏里的 NPC，  
> 需要从左下角出发，前往右上角目标点，  
> 途中要绕开中间区域不断变化的障碍物。

这里不是做“固定地图背题”。

训练时，障碍物位置会在一组对角线锚点附近**随机采样**。  
推理时，每次运行也会重新生成一组**新的随机障碍物布局**。

这样更接近真实场景：

- 起点和终点是稳定任务目标
- 障碍物会变化
- 模型学到的应该是“绕障去目标”的能力，而不是记住某一张固定地图

## 这个 demo 想说明什么

这个 demo 不是为了单纯展示“嵌套结构很复杂”。

它要表达的是：

1. 可以把一个导航问题拆成多个子职责
2. 不同子网可以并行处理同一份输入
3. 上层子网再把多个子结果融合成最终动作
4. 这种带嵌套关系的网络，已经可以按项目主流程完成：
   - 生成
   - 训练
   - 推理
   - CLI 可视化验证

这个 demo 的层次关系是：

- 根容器：`agent`
- 子容器：`perception`
  - `target_encoder`
  - `obstacle_encoder`
- 子容器：`planner`
  - `fusion_head`

也就是：

`[target_encoder, obstacle_encoder] -> fusion_head`

## 训练数据是怎么生成的

训练标签不是手写的固定方向，也不是简单朝目标直冲。

这里用的是：

- **随机障碍物场景生成**
- **A\* 网格路径规划**
- **线段级碰撞检查**

也就是说，训练集里的“下一步应该往哪走”，
是由一个会检查障碍物占用和穿越碰撞的参考规划器生成的。

它的目标就是保证：

- 不穿过障碍物
- 不把对角穿角当成合法路径
- 真正学的是“绕开障碍，再继续接近目标”

## 输入和输出

- 输入：
  - `current_x current_y`
  - `target_x target_y`
  - `obstacle1_x obstacle1_y`
  - `obstacle2_x obstacle2_y`
  - `obstacle3_x obstacle3_y`
- 输出：
  - `move_dx move_dy`

## CLI 输出怎么看

推理程序不是只打印一组坐标。

它会把整个移动过程按帧打印成字符地图。
CLI 会同时打印：

- 模型原始预测方向
- 如果原始方向会撞障碍物或明显不适合推进，执行器选择的安全步进方向

符号说明：

- `o`：已经走过的轨迹
- `C`：当前位置
- `T`：目标位置
- `X`：障碍物区域
- `^ v < > *`：模型预测的下一步方向
- `!`：下一步会撞障碍物，这是错误结果

所以你可以直接看出：

- 它是不是在朝目标移动
- 它是不是在绕障
- 它是不是试图穿过障碍物

这个“安全步进”不是为了掩盖模型问题，而是为了更接近真实部署：

- 模型负责提出动作倾向
- 执行器负责最后一道障碍物占用检查
- 最终展示的是“模型 + 安全执行”在随机场景下的实际导航过程

## 运行流程

这个 demo 仍然严格遵循统一的 6 步流程：

1. 编译 generate
2. 运行 generate
3. 编译 train
4. 运行 train
5. 编译 infer
6. 运行 infer

`run_demo.bat` / `run_demo.sh` 会按这个顺序自动执行。  
其中第 6 步不再读取固定障碍物输入，而是直接运行一次随机场景推理。
