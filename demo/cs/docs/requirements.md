# CS Demo 需求与范围定义

## 1. 文档目标

把用户的原始想法整理成一个可执行、可收敛、适合本仓库主流程的 demo 需求定义。

## 2. 原始目标复述

用户希望新增一个更强的 demo，用来推动项目成熟。  
原始效果预期为：

1. 程序启动时联动启动 CS1.6
2. 用户在游戏中手动完成开局
3. 用户在命令行输入一个地点标识符
4. 系统自动截图并分析当前画面
5. 系统通过模拟键盘输入控制人物移动
6. 角色逐步接近目标点

## 3. 需求收敛结论

为了让方案可控、可验证、可按仓库主流程落地，建议把第一版需求收敛为：

> **“本地离线/私有测试环境中的视觉导航 demo”**

即：

- 游戏环境限定为用户自有、可控、允许自动化测试的本地环境
- 第一阶段只做**导航与移动**
- 暂不做瞄准、射击、压枪、跟枪、敌我识别对抗
- 暂不做公开多人房间自动化
- 地图固定为 `de_dust2`
- 目标表达固定为**唯一地点标识符**

## 4. 强制边界

### 4.1 使用边界

本 demo 设计仅面向：

- 单机练习
- 本地 bot 房
- 私有测试环境
- 经用户明确授权的自测环境

本 demo 设计**不面向**：

- 公开多人对战房间
- 排位或竞技对战
- 破坏游戏公平性的自动化对抗

### 4.2 技术边界

本阶段不纳入：

- 反作弊绕过
- DLL 注入
- 内存读取
- 网络封包控制
- 非公开接口逆向

### 4.3 任务边界

本阶段只做：

- 启动/附着游戏窗口
- 自动截图
- 基于视觉或视觉+规则估计当前位置/朝向/局部状态
- 根据命令行输入的目标地点标识符生成移动动作
- 通过标准系统输入接口模拟键盘控制

## 5. 第一版 demo 的任务定义

### 5.1 用户故事

作为 demo 使用者，我希望：

1. 启动 `demo/cs` 的推理程序
2. 看到程序尝试启动或附着到 CS1.6 窗口
3. 在游戏里手动进入 `de_dust2` 并完成开局
4. 在命令行输入一个目标地点标识符
5. 程序开始自动观察画面并控制角色移动
6. 我可以在命令行中看到当前状态、目标、预测动作、执行结果
7. 我可以随时中止控制

### 5.2 输入定义

第一版输入不再使用数值坐标，而使用 `de_dust2` 的**地点标识符**：

- `target_place_id`

示例：

- `a_site`
- `b_site`
- `mid`
- `long_doors`
- `catwalk`
- `ct_spawn`
- `upper_tuns`

输入规则建议为：

- 每个地点只有一个合法标识
- 标识必须明确、唯一、简短、可读
- 不做别名兼容
- 不做语义理解
- 不做大小写宽松匹配
- 命令层先把标识符转换为内部整数 ID 或 one-hot，再传给网络层

### 5.3 输出定义

推理程序的主要输出为：

- 当前估计位置
- 当前估计朝向
- 当前目标地点标识
- 当前所在区域
- 当前动作决策
- 执行状态
- 成功/失败/中止原因

动作集合建议先限制为：

- `forward`
- `backward`
- `move_left`
- `move_right`
- `turn_left_small`
- `turn_right_small`
- `stop`

## 6. 功能需求

### FR-1 游戏联动

推理程序需要具备以下二选一能力：

- 启动 CS1.6
- 或附着到已运行的 CS1.6 窗口

### FR-2 运行前检查

推理程序在开始控制前，必须完成基础检查：

- 找到目标窗口
- 校验窗口尺寸或采样区域可用
- 校验当前地图属于已支持集合
- 校验控制焦点正确

任一失败应快速报错并停止。

### FR-3 目标输入

程序应支持用户在命令行输入目标地点标识符，并允许：

- 开始前输入
- 到达一个地点后继续输入下一个地点
- 用户主动取消当前目标
- 输入非法地点标识符时快速报错

### FR-4 画面采集

程序应周期性采集游戏窗口图像，用于：

- 位置估计
- 朝向估计
- 局部场景识别
- 进度判断

### FR-5 状态估计

程序需要把截图转换为一个可供控制器消费的状态表示，至少包括：

- 当前估计位置
- 当前估计朝向
- 当前所在 callout 区域
- 目标相对方向
- 当前位置到目标地点区域的误差

### FR-6 动作推理

程序需要根据当前状态与目标状态，输出一步控制动作。

### FR-7 输入执行

程序需要通过标准系统输入方式执行键盘动作，并保证：

- 有明确按下/释放时机
- 可以安全停机
- 不出现卡键

### FR-8 可观测性

程序需要输出足够的调试信息，至少包括：

- 当前帧编号
- 当前目标
- 当前位置估计
- 动作预测
- 动作执行结果
- 失败原因

### FR-9 人工接管

程序必须提供人工接管机制，例如：

- 命令行停止命令
- 指定热键急停
- 窗口失焦立即停机

## 7. 非功能需求

### NFR-1 对齐仓库主流程

未来实现时必须严格遵循仓库定义的主链路：

1. 第一次编译：启用网络类型并构建生成器/相关组件
2. 第一次运行：调用 profiler 生成训练/推理代码
3. 第二次编译：训练工程
4. 第三次编译：推理工程

### NFR-2 demo 目录组织

未来实现时，`demo/cs` 应与现有 demo 一致，采用清晰拆分结构，至少预留：

- `demo/cs/generate/`
- `demo/cs/train/`
- `demo/cs/infer/`
- `demo/cs/docs/`

### NFR-3 平台约束

未来实现必须符合当前仓库环境：

- Windows 11
- PowerShell 命令语法
- clang 工具链
- 构建前先进入 `vcvars64.bat` 环境

### NFR-4 质量约束

未来实现必须满足仓库既有质量要求：

- C99
- 零警告
- 警告即错误
- 注释量不少于 50%
- 首错即停

### NFR-5 运行约束

第一版目标应偏保守，优先保证：

- 可复现
- 可解释
- 可停机
- 可记录

而不是追求复杂对战能力。

### NFR-6 版本推进约束

`demo/cs` 后续开发必须采用单功能串行版本制：

- 一个版本只实现一个明确功能模块
- 当前版本必须先开发完成并测试完成
- 当前版本未验收通过前，不进入下一个版本
- 禁止把多个核心模块打包进同一个首版
- 每个版本都必须有独立的输入、输出、测试口径与完成定义

## 8. 建议的范围切分

不再把“第一版”定义成完整导航闭环，而是拆成多个串行版本。

### 第一阶段（推荐）

固定地图 `de_dust2`、固定分辨率、本地 bot 房、单地点导航。

### 第二阶段

支持多个地点、连续 waypoint、基础绕障。

### 第三阶段

支持更复杂地图状态、一定程度的视觉扰动与场景变化。

### 暂不建议纳入

- 战斗决策
- 自动开火
- 敌人识别对抗
- 公开多人对战自动化

## 9. 验收口径

每个版本单独验收，不再用一个“大而全”的第一版一次性验收所有能力。

第一版 demo 可以认为成功，至少需要满足：

1. 可在受支持的本地测试环境中启动或附着游戏窗口
2. 用户可输入 `de_dust2` 的支持地点标识符
3. 程序可持续采集画面并完成状态估计
4. 程序可持续输出移动动作
5. 角色可在 `de_dust2` 上稳定接近目标地点区域
6. 程序具备急停与失败回退能力
7. 运行过程有可读日志

## 10. 当前建议的关键决策

### 决策 A：先做导航，不做战斗

原因：更适合作为本仓库“复杂感知 + 闭环控制”的成熟度 demo。

### 决策 B：目标先做唯一地点标识符，不做世界坐标

原因：命令接口更稳定，网络输入更直接，避免别名和语义歧义。

### 决策 C：第一版只做 `de_dust2`

原因：地图熟悉度高、地点称呼成熟、便于构建标准化路径与验收。

### 决策 D：允许“规则模块 + 神经网络模块”混合

原因：当前项目强调流程与结构治理，第一版不应把所有问题都压给网络端。

### 决策 E：采用单模块串行版本推进

原因：降低首版范围，减少耦合，便于测试、回归和问题定位。

## 11. 待确认问题

以下问题如果后续进入开发，应先与用户确认：

1. 第一版支持哪些 `de_dust2` 地点标识符
2. 每个地点标识符对应的目标区域如何定义
3. 是否接受“仅支持固定分辨率/固定窗口模式”
4. 第一版是否允许只支持本地 bot 房
5. 第一版是否允许只控制移动，不控制视角
