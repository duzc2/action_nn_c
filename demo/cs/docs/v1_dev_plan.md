# Version 1 开发计划

## 1. 文档目标

本文档用于把 Version 1 真正落成一个可开发、可测试、可关闭的工作计划。

Version 1 只做：

> **区域识别**

所以本计划只围绕以下闭环展开：

- 数据采集
- 数据整理
- 区域识别训练
- 区域识别推理
- 测试与结果记录

## 2. 开发原则

Version 1 必须继续遵守单模块串行原则：

- 只做区域识别
- 不夹带朝向识别
- 不夹带路线规划
- 不夹带动作控制

并且继续遵守“当前阶段完成后再进入下一阶段”。

## 3. Version 1 的完成定义

只有满足以下条件，才算 Version 1 完成：

1. 采集工具能稳定产出原始 session
2. 标注工具能稳定产出分段标签
3. 数据构建工具能稳定生成 train / val / test
4. 训练程序能基于该数据集完成训练
5. 推理程序能对单帧或实时截图输出区域标签
6. 有完整测试结果与失败样本记录

## 4. 推荐开发顺序

建议按下面顺序推进，不要并行乱做。

## Step 1：固定 Version 1 配置

目标：

- 固定区域标签集合
- 固定分辨率
- 固定地图
- 固定数据目录

产物：

- 标签清单文档
- 数据目录约定
- place token 字典

完成标准：

- 文档明确
- 不再频繁改动标签集合

## Step 2：做 `cs_capture_session`

目标：

- 能附着窗口
- 能持续截图
- 能保存 session

最小成功标准：

- 能生成 `session.json`
- 能落盘 `frames/`
- 能正确停止采集

此阶段不要求：

- 标签功能
- 数据切分功能

## Step 3：做 `cs_label_session`

目标：

- 能按用户切换记录 `place_token`
- 能输出 `label_segments.json`

最小成功标准：

- `set` 能开启新标签段
- 切换 token 能正确关闭上一个段
- session 结束后标签文件结构正确

## Step 4：联调采集 + 分段标注

目标：

- 做出第一批真实 raw session

最小成功标准：

- 至少 1 个 session 可完整采下来
- 至少覆盖 2~3 个区域
- 标签段和帧号可对上

这一步非常重要，因为它是后续所有开发的真实输入基础。

## Step 5：做 `cs_dataset_build`

目标：

- 从 raw session 自动生成训练数据集清单

最小成功标准：

- 能读 `session.json`
- 能读 `label_segments.json`
- 能产出 `train_list.json`
- 能产出 `val_list.json`
- 能产出 `test_list.json`

建议这一步就加入：

- 最小间隔抽样
- 去重
- 坏帧过滤

## Step 6：做 `cs_dataset_report`

目标：

- 让用户知道数据够不够、哪里不够

最小成功标准：

- 能输出每类样本数
- 能输出 session 覆盖情况
- 能提示类别失衡

## Step 7：完成第一批数据采集

目标：

- 获取第一批可训练数据

建议范围：

- 先采 5 类：
  - `t_spawn`
  - `ct_spawn`
  - `mid`
  - `a_site`
  - `b_site`

最小成功标准：

- 每类都有样本
- train / val / test 已生成
- 数据报告可读

## Step 8：做 Version 1 generate

目标：

- 定义区域识别网络结构
- 生成训练/推理代码

首选网络：

- `cnn + mlp`

可选增强：

- `cnn + rnn + mlp`

最小成功标准：

- generate 可跑通
- 生成代码可进入训练与推理工程

## Step 9：做 Version 1 train

目标：

- 训练区域识别模型

最小成功标准：

- 可读取 `train_list.json`
- 可正常训练
- 能输出权重 `.bin` 与权重 `.c`

此时先不追求最好效果，先追求：

- 流程打通
- 能收敛
- 能评估

## Step 10：做 Version 1 infer

目标：

- 读取单帧图片或实时截图
- 输出区域标签与置信度

最小成功标准：

- 能对离线图片推理
- 能输出标签
- 日志可读

## Step 11：离线评估

目标：

- 判断区域识别到底是否成立

必须输出：

- 总准确率
- 每类准确率
- 混淆矩阵
- 典型失败样本

## Step 12：在线验证

目标：

- 在真实 `de_dust2` 运行环境中实时输出区域预测

注意：

- 这里只做“识别”
- 绝不进入动作控制

最小成功标准：

- 能连续输出预测结果
- 不崩溃
- 日志可回看

## 5. 每一步的测试门禁

每一步都必须有“过关标准”。

### Gate 1：采集器门禁

通过条件：

- 能稳定截图 1 个 session
- session 文件结构正确

### Gate 2：标注器门禁

通过条件：

- 标签段切换正确
- 段边界可追溯

### Gate 3：数据构建门禁

通过条件：

- 能稳定产出 train / val / test
- 无路径错误
- 样本字段完整

### Gate 4：训练门禁

通过条件：

- 能完成训练
- 能产出权重
- 无结构错误

### Gate 5：推理门禁

通过条件：

- 能对输入图输出区域标签
- 输出格式稳定

### Gate 6：评估门禁

通过条件：

- 有明确评估结果
- 有失败样本记录
- 能得出是否继续的结论

## 6. 推荐的开发里程碑

### M1：采集链路打通

范围：

- `cs_capture_session`
- `cs_label_session`

标志：

- 能采 session
- 能写标签段

### M2：数据集链路打通

范围：

- `cs_dataset_build`
- `cs_dataset_report`

标志：

- 能生成可训练数据集

### M3：训练推理链路打通

范围：

- generate
- train
- infer

标志：

- 能完成区域识别训练和推理

### M4：评估闭环完成

范围：

- 离线评估
- 在线验证
- 失败样本记录

标志：

- 能判断 Version 1 是否合格

## 7. 当前最推荐先开发的内容

如果现在真的要开始动手，不要先碰网络。

最推荐先做：

> **先做 `cs_capture_session` 和 `cs_label_session`。**

原因：

- 没有稳定数据，就没有后续训练
- 先做网络没有意义
- 工具链稳定后，后面的工作更容易推进

## 8. 什么时候可以进入 Version 2

只有在以下条件成立时，才建议进入 Version 2：

1. Version 1 数据链路稳定
2. Version 1 训练链路稳定
3. Version 1 推理链路稳定
4. 已有测试结果
5. 已确认区域识别结果足以作为下一版输入

否则，不建议进入 Version 2。
