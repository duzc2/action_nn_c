# Version 1 数据采集工具方案

## 1. 文档目标

本文档专门回答：

- Version 1 需要提供哪些工具
- 这些工具分别自动化什么
- 用户具体怎么操作

目标是让用户只做最简单的行为，而把大量重复工作交给工具。

## 2. 设计原则

Version 1 的工具必须满足：

- 用户操作简单
- 自动化比例高
- 主流程可通过 GUI 完成
- 不要求用户逐帧标注
- 结果可复盘
- 失败可定位

## 3. 必须提供的工具

## 3.1 工具 1：窗口采集器 `cs_capture_session`

功能：

- 查找并附着 CS1.6 窗口
- 定时截图
- 保存原始帧
- 记录 session 元数据

用户只需要：

- 打开游戏并进入地图

## 3.2 工具 2：分段标注器 `cs_label_session`

功能：

- 接收当前区域 token
- 记录标签切换时间点
- 输出分段标签文件

用户只需要：

- 当进入新区域时按一次快捷键或输入一次 token

## 3.3 工具 3：数据集构建器 `cs_dataset_build`

功能：

- 读取原始帧
- 读取分段标签
- 自动抽样
- 自动去重
- 自动过滤坏帧
- 自动生成 train / val / test 清单

用户不需要手工处理图片。

## 3.4 工具 4：数据报告器 `cs_dataset_report`

功能：

- 输出每类样本数
- 输出 session 覆盖情况
- 输出数据是否失衡
- 给出“还缺哪些区域数据”的提示

## 4. 推荐的用户工作流

## 4.1 启动前

用户：

1. 打开 CS1.6
2. 按固定设置进入 `de_dust2`

工具：

1. 检测窗口
2. 校验分辨率
3. 校验地图

## 4.2 开始采集

用户：

1. 执行采集工具
2. 输入当前区域 token，例如 `mid`

工具：

1. 开始截图
2. 记录当前标签段

## 4.3 自由移动

用户：

- 在当前区域内无目的走动和转头

工具：

- 自动连续截图
- 自动写盘
- 自动记录帧号和时间戳

## 4.4 切换区域

用户：

- 到达新区域后输入新的 token，例如 `a_site`

工具：

- 自动结束上一段
- 自动开始新标签段

## 4.5 结束

用户：

- 停止采集

工具：

- 保存 session 结果
- 输出采集摘要
- 可直接调用 dataset build

## 5. 推荐的交互形式

Version 1 需要 GUI。  
推荐采用：

- GUI 作为主入口
- CLI 作为脚本入口
- 文本日志作为补充

原因：

- 采集过程持续时间较长，用户需要看到实时帧数和状态
- 纯后台工具不利于判断是否仍在正常工作
- 标签切换、session 路径和错误信息需要直接显示
- 数据集构建和报告生成更适合表单式操作

建议 GUI 至少覆盖：

- 当前 session
- 当前状态
- 当前 `place_token`
- 已采集帧数
- 当前输出目录
- 最近错误信息

例如：

```text
capture start
label set mid
label set a_site
capture stop
```

或者：

- `F1` 开始/停止采集
- `F2` 输入当前 token

但这些 CLI 或热键入口应视为 GUI 的补充，不应替代 GUI。

## 6. 应自动化的关键能力

以下能力建议由工具自动完成。

## 6.1 自动截图

避免用户自己录屏、自己截帧。

## 6.2 自动标签分段

用户只提供“当前区域已切换”的事件，  
工具自动把这个标签应用到一段连续帧。

## 6.3 自动数据瘦身

工具自动做：

- 帧抽样
- 重复帧删除
- 坏帧过滤

## 6.4 自动数据切分

工具自动生成：

- `train_list.json`
- `val_list.json`
- `test_list.json`

## 6.5 自动报告

工具自动告诉用户：

- 哪类样本太少
- 哪个 session 标注异常
- 哪些区域还需要补采

## 7. 用户不应该做的事

不应该要求用户：

- 逐帧截图
- 逐帧写标签
- 手工整理训练集目录
- 手工统计每类样本数
- 手工去重

这些都应该由工具完成。

## 8. 推荐的开发优先级

如果后续开始真正开发 Version 1，工具优先级建议如下：

### Priority 1

- `cs_capture_session`
- `cs_label_session`

### Priority 2

- `cs_dataset_build`

### Priority 3

- `cs_dataset_report`

原因：

- 没有前两个工具，就没有稳定原始数据
- 没有 `dataset_build`，训练很难重复
- `report` 重要，但可以稍后补

## 9. 对 Version 1 成败的影响

Version 1 是否能顺利推进，关键不只是网络结构，  
更关键的是：

> **有没有一套用户能真正执行的数据采集工具链。**

所以 Version 1 的“必要工具”不是附属品，而是主体工作的一部分。
