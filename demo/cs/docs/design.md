# CS Demo 总体设计构思

## 1. 设计目标

本 demo 的价值，不是做一个“游戏外挂”，而是做一个能体现本项目成熟度的复杂闭环示例：

- 有真实视觉输入
- 有结构化目标输入
- 有神经网络参与决策
- 有外部环境反馈
- 有可观测、可中止、可复现实验流程

因此建议把它定义为：

> **“面向本地测试环境、基于 de_dust2 唯一地点标识符目标的视觉导航控制 demo”**

## 2. 推荐演示效果

不建议把首版直接定义为完整导航闭环。  
建议改为：

> **每一个版本只完成一个功能模块，并单独形成可测试 demo。**

建议顺序：

1. Version 1：区域识别
2. Version 2：朝向与局部运动状态识别
3. Version 3：路线规划
4. Version 4：目标编码与控制输入准备
5. Version 5：动作策略影子模式
6. Version 6：闭环移动控制

这样每个版本都有明确输入、明确输出、明确测试边界。

推荐把第一版演示效果定义为：

1. 用户启动 `cs_infer`
2. 程序启动或附着 CS1.6
3. 用户手动进入 `de_dust2` 并完成开局
4. 用户在命令行输入目标地点标识符，例如 `a_site`
5. 程序持续截图、分析、决策、执行移动
6. 角色逐步接近目标点
7. CLI 持续显示当前位置估计、目标误差、动作选择、运行状态

## 3. 与仓库主流程的对应关系

未来若进入开发，建议严格映射到现有 demo 的三段式结构：

### 3.1 generate 阶段

职责：

- 定义 `demo/cs` 使用的网络拓扑
- 指定 profiler 输出路径
- 生成训练/推理代码与固定头文件

### 3.2 train 阶段

职责：

- 读取训练样本
- 训练导航控制网络
- 输出权重 `.bin` 与权重 `.c`

### 3.3 infer 阶段

职责：

- 启动/附着游戏
- 采集画面
- 加载网络权重
- 执行闭环控制

## 4. 建议的系统分层

建议把未来实现分成 6 层：

### 4.1 进程与窗口编排层

负责：

- 启动或附着游戏进程
- 查找窗口
- 维护焦点与生命周期

### 4.2 画面采集层

负责：

- 定时截图
- 裁剪有效区域
- 统一图像尺寸与颜色格式

### 4.3 状态理解层

负责从画面中提取控制需要的状态，例如：

- 当前位置估计
- 朝向估计
- 局部可通行区域
- 当前所处地点区域估计
- 与目标地点的相对关系

### 4.4 控制决策层

负责把状态映射成一步动作：

- 前进
- 后退
- 平移
- 小角度转向
- 停止

### 4.5 输入执行层

负责：

- 键盘按下/释放
- 安全超时
- 急停

### 4.6 观测与日志层

负责：

- CLI 状态输出
- 错误记录
- 数据回放辅助

## 5. 推荐的网络拓扑思路

为了体现项目“已有基础网络类型 + 具备组织复杂网络能力”的价值，建议 `demo/cs` 不只放一个单 MLP，而是设计成一个分层组合网络。

推荐拓扑如下：

```text
agent
├─ perception_group
│  ├─ frame_encoder
│  └─ pose_estimator
└─ control_group
   ├─ callout_encoder
   └─ action_head
```

### 5.1 frame_encoder

候选类型：

- `cnn`
- `tiny_tcn`
- 小型 `mlp`（仅在输入被强力预处理时）

职责：

- 从截图中提取压缩视觉特征

### 5.2 pose_estimator

候选类型：

- `mlp`
- `rnn`

职责：

- 结合连续多帧特征，估计当前位置与朝向

### 5.3 callout_encoder

候选类型：

- `mlp`

职责：

- 把用户输入的地点标识符、当前所在区域、相对目标关系编码成控制特征

### 5.4 action_head

候选类型：

- `mlp`
- `rnn`

职责：

- 输出当前一步动作
- 让动作具备一定时序稳定性

## 6. 推荐的“网络 + 规则”混合方案

第一版不建议把所有问题都交给神经网络。

更稳妥的方案是：

- 网络负责：感知压缩、位姿估计、动作倾向
- 规则负责：窗口检查、焦点检查、急停、超时、动作裁剪、到达判定

这样更符合仓库既有原则：

- 优先通过流程和结构治理问题
- 不把所有稳定性问题都转化成训练问题

## 7. 运行时闭环流程

推荐闭环如下：

1. 推理程序启动
2. 启动或附着游戏窗口
3. 用户手动进入测试地图
4. 用户输入目标地点标识符
5. 采集当前画面
6. 预处理画面
7. 估计当前位置与朝向
8. 解析目标地点并计算目标相对误差
9. 推理一步动作
10. 通过键盘执行短时动作
11. 再次截图确认是否逼近目标
12. 达到阈值则停止，否则继续循环

## 8. 数据与训练构思

### 8.1 样本单位

建议把一条训练样本定义为：

- 当前截图或连续多帧截图
- 当前估计位置/标签位置
- 当前 callout 区域标签
- 当前朝向
- 目标地点标识符
- 教师动作标签
- 可选的动作执行结果

### 8.2 标签来源

建议优先考虑以下来源：

- 人工示教轨迹
- 固定地图上的 waypoint 教师策略
- 截图 + 人工标注位置

第一版不建议依赖复杂自动标注体系。

### 8.3 训练目标

第一版更适合做：

- 分类式动作预测

而不是：

- 直接回归复杂连续控制量

因为分类动作更容易解释、裁剪和回退。

## 9. 推荐的目标表示定义

根据当前需求，第一版目标不再是数值坐标，而是 `de_dust2` 的地点标识符。

推荐内部表示为两层：

### 9.1 用户输入层

- `a_site`
- `b_site`
- `mid`
- `long_doors`
- `catwalk`
- `ct_spawn`

### 9.2 系统内部层

每个地点标识符映射到：

- 一个标准 callout ID
- 一个目标区域锚点
- 一个可选 waypoint 集

即：

> **命令行只接受唯一标识符；进入控制逻辑前先转成整数化目标表示。**

这样可以保证输入层稳定、唯一、易校验。

## 10. 支持地图策略

第一版建议只支持：

- `de_dust2` 单地图起步
- 少量固定出生/测试位置
- 固定窗口模式
- 固定分辨率

不建议第一版追求：

- 任意地图
- 任意 HUD
- 任意分辨率
- 任意视角条件

## 11. 安全与停机设计

无论后续怎么实现，以下机制都应被视为强制项：

- 窗口失焦立即停机
- 目标窗口丢失立即停机
- 达到动作超时立即停机
- 用户急停热键立即释放全部按键
- 地图不匹配立即拒绝运行

## 12. 建议的阶段规划

在当前新策略下，原有 “Phase 0 / Phase 1 / Phase 2” 更适合作为完整闭环远期目标，  
真正实施时建议改为参考 `module_versions.md` 的串行版本推进。

### Phase 0：影子模式

只做：

- 启动/附着
- 截图
- 状态估计
- 打印建议动作

**不真正发送控制输入。**

价值：

- 先验证视觉链路是否成立

### Phase 1：单目标导航

做：

- 固定地图
- 单目标点
- 闭环移动
- 键盘控制

这是最推荐的第一版正式 demo。

### Phase 2：多 waypoint 导航

做：

- 多目标连续输入
- 路径分段执行
- 更稳定的控制逻辑

### Phase 3：更复杂场景

可选扩展：

- 轻量绕障
- 更复杂的出生点变化
- 更强的多帧时序建模

## 13. 当前最值得落地的演示版本

如果目标是“尽快做出一个能明显推动项目成熟的 demo”，我建议后续实现时优先落这个版本：

> **固定地图 de_dust2 + 本地 bot 房 + 单目标地点标识符 + 自动截图 + 自动移动 + CLI 可观测日志**

理由：

- 复杂度足够高，能体现项目能力
- 边界足够清楚，能控制风险
- 容易拆成 generate/train/infer 三段
- 便于后续逐步升级

## 14. 当前不建议直接做的版本

暂不建议直接把第一版定义为：

- 公开多人房间自动行动
- 边导航边战斗
- 自动瞄准与射击
- 泛化到多地图多分辨率

这样会同时放大：

- 数据难度
- 控制难度
- 稳定性风险
- 验证难度

## 15. 进入开发前建议确认的事项

1. 第一版支持的 callout 标识符列表
2. 每个 callout 标识符的目标区域定义
3. 是否允许固定窗口分辨率
4. 是否接受 Phase 0 先做影子模式
5. 是否接受第一版只做导航闭环
