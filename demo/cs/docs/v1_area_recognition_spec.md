# Version 1 区域识别规格说明

## 1. 文档目标

本文档只定义 `demo/cs` 的 **Version 1：区域识别**。

这一版只做一件事：

> **输入 `de_dust2` 截图，输出当前区域标签。**

不做：

- 朝向识别
- 局部运动状态识别
- 路线规划
- 动作策略
- 键盘控制
- 闭环导航

## 2. Version 1 的定位

Version 1 是整个 `demo/cs` 的基础版。

它的意义不是“已经能导航”，而是先回答一个最基础的问题：

> **系统能不能稳定识别当前在地图的哪个区域。**

如果这一步不稳定，后续：

- 朝向判断
- 路线规划
- 动作控制

都没有可靠输入基础。

## 3. 功能定义

### 3.1 输入

输入为一张截图，来源于：

- `de_dust2`
- 固定窗口模式
- 固定分辨率
- 固定画面设置

建议最小输入结构：

- `frame_path` 或内存图像
- 可选 `capture_timestamp`

### 3.2 输出

输出为：

- `predicted_place_id`
- `predicted_place_token`
- `confidence`（建议保留）

### 3.3 最小目标

给定一张截图，系统能输出当前区域属于哪个已定义区域标签。

## 4. Version 1 推荐标签范围

第一版不要贪多，建议只做**高辨识度大区域**。

推荐先用以下 8 个标签：

| place_id | place_token |
|---:|---|
| 0 | `t_spawn` |
| 1 | `ct_spawn` |
| 2 | `mid` |
| 3 | `a_site` |
| 4 | `b_site` |
| 5 | `long_doors` |
| 6 | `catwalk` |
| 7 | `upper_tuns` |

原因：

- 语义清楚
- 区域较大
- 数据容易采
- 混淆相对可控

不建议 Version 1 就纳入：

- `goose`
- `a_car`
- `b_window`
- `back_plat`
- `lower_mid`

这类更细碎、更容易混淆的点位。

## 5. 数据采集规范

更具体的数据生成、标注和工具方案，见：

- `v1_dataset_spec.md`
- `v1_data_collection_tooling.md`

### 5.1 采集环境必须固定

建议固定以下条件：

- 地图：`de_dust2`
- 分辨率：固定一种
- 窗口模式：固定一种
- FOV：固定
- HUD：固定
- 亮度：固定
- 贴图质量：固定
- 角色阵营：尽量固定采集规则

### 5.2 每条样本建议字段

每条样本至少记录：

- `image_path`
- `place_id`
- `place_token`
- `session_id`
- `capture_index`

建议附加：

- `player_team`
- `view_variant`
- `notes`

### 5.3 采集要求

每个区域建议覆盖：

- 多个站位点
- 多个朝向
- 多个微小位移
- 多张截图

也就是说：

> 不要让“区域识别”退化成“记住一个固定机位截图”。

### 5.4 数据划分建议

建议按采集批次切分：

- 训练集
- 验证集
- 测试集

避免同一站位连续截图同时落入训练和测试。

## 6. 网络结构建议

Version 1 优先从最简单、最稳妥的结构开始。

### 6.1 首选结构

```text
image -> cnn -> mlp -> place_class
```

原因：

- 任务本质是图像分类
- 当前仓库已有 `cnn` 与 `mlp`
- 实现复杂度最低

### 6.2 可选增强结构

如果单帧分类不稳定，再升级为：

```text
frame_seq -> cnn -> rnn -> mlp -> place_class
```

但这不应该作为 Version 1 默认方案。

建议策略：

1. 先做 `cnn + mlp`
2. 先测单帧是否够用
3. 不够再加 `rnn`

## 7. generate / train / infer 的职责

Version 1 也必须遵循仓库主流程。

### 7.1 generate

职责：

- 定义区域识别网络
- 指定 profiler 输出路径
- 只生成 Version 1 所需训练/推理代码

### 7.2 train

职责：

- 读取区域识别数据集
- 训练分类模型
- 输出权重 `.bin` 与权重 `.c`

### 7.3 infer

职责：

- 读取一张截图或实时截图
- 运行区域分类推理
- 输出预测标签与置信度

## 8. 测试设计

### 8.1 离线测试

Version 1 必须先完成离线测试。

至少输出：

- 总准确率
- 每类准确率
- 混淆矩阵

### 8.2 在线测试

在线测试只做：

- 采实时截图
- 输出当前预测区域
- 打印日志

不做任何动作执行。

### 8.3 失败分析

每次测试至少保留：

- 预测错误样本
- 真实标签
- 预测标签
- 置信度

用于后续分析：

- 是否标签定义有问题
- 是否区域过细
- 是否样本覆盖不足

## 9. 验收标准

Version 1 是否完成，不看“能不能导航”，只看区域识别本身。

建议完成标准：

1. 已支持的区域标签集合固定
2. 数据集结构固定
3. 训练流程可重复执行
4. 推理程序可输出区域标签
5. 有独立测试集评估结果
6. 有失败样本记录
7. 有明确结论：是否进入 Version 2

## 10. 建议的完成定义

只有满足以下条件，才算 Version 1 真正完成：

- 文档完成
- 数据完成
- 训练完成
- 测试完成
- 结果记录完成

缺任何一项，都不建议进入下一版。

## 11. Version 1 完成后再做什么

Version 1 通过后，优先进入：

> **Version 2：朝向与局部运动状态识别**

因为这两个能力仍然属于“感知层”，和区域识别连续性最好，耦合也最小。
