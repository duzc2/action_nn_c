# Version 1 风险与应对

## 1. 文档目标

本文档专门整理 Version 1 的常见失败点。

Version 1 只做：

> **区域识别**

所以这里的风险分析也只围绕：

- 数据采集
- 标签质量
- 数据集构建
- 模型训练
- 推理验证

## 2. 使用原则

遇到问题时，优先按以下顺序排查：

1. 先查流程
2. 再查数据
3. 再查标签
4. 再查工具输出
5. 最后才查网络结构

也就是说：

> **不要一看到效果不好，就第一反应去改模型。**

很多问题其实出在：

- 采集条件不一致
- 标签切换错误
- 数据集失衡
- 清洗规则有问题

## 3. 主要风险总表

| 风险编号 | 风险 | 影响 |
|---|---|---|
| R1 | 游戏设置不一致 | 数据分布漂移，模型难学 |
| R2 | 标签切换滞后或错误 | 训练样本污染 |
| R3 | 区域定义过细 | 类间混淆严重 |
| R4 | 数据重复太多 | 训练集看似很大，实际无效 |
| R5 | 类别失衡严重 | 模型偏向高频类别 |
| R6 | 坏帧太多 | 降低有效样本比例 |
| R7 | 字典不一致 | token / id 混乱 |
| R8 | 训练集和测试集泄漏 | 指标虚高 |
| R9 | 在线推理不稳定 | 实际不可用 |
| R10 | 过早引入复杂结构 | 排查困难，范围失控 |

## 4. 详细风险说明

## 4.1 R1：游戏设置不一致

### 现象

- 同一类别在不同 session 中视觉差异很大
- 模型训练不稳定
- 在线效果明显比离线差

### 常见原因

- 分辨率变了
- 亮度变了
- HUD 变了
- 窗口模式变了
- 画面参数变了

### 排查方法

1. 检查 `session.json`
2. 比较不同 session 的分辨率和参数
3. 抽看样本图像

### 应对策略

- 固定采集设置
- 工具启动时校验分辨率
- 不符合设置时拒绝采集

## 4.2 R2：标签切换滞后或错误

### 现象

- 某些类别边界样本明显错标
- 混淆矩阵中出现不合理对冲
- 同一区域预测总偏到相邻区域

### 常见原因

- 用户进入新区后没有及时切 token
- 用户尚未进入新区就提前切 token

### 排查方法

1. 回看 `label_segments.json`
2. 抽查切换边界附近帧
3. 检查失败样本是否集中在边界段

### 应对策略

- 提醒用户“确认进入新区域后再切 token”
- 边界附近帧可在 build 阶段降低采样权重或剔除
- 初期优先采大区域，减少边界歧义

## 4.3 R3：区域定义过细

### 现象

- 某些类别准确率长期很低
- 两类区域几乎总互相混淆
- 用户自己也难以稳定区分

### 常见原因

- 一开始就纳入太细的点位
- 标签语义边界不清

### 排查方法

1. 看混淆矩阵
2. 看失败样本是否集中在细粒度类别
3. 问一个简单问题：人类看图能不能稳定分开

### 应对策略

- 先合并成大区域
- Version 1 只保留高辨识度标签
- 细粒度点位留到后续版本

## 4.4 R4：数据重复太多

### 现象

- 训练集样本数看着很多
- 但有效提升很小
- 模型泛化差

### 常见原因

- 连续帧太密
- 同一站位同一视角保存太多

### 排查方法

1. 看 build 报告中的去重比例
2. 随机抽看连续样本
3. 检查 `min-frame-step` 是否太小

### 应对策略

- 提高抽样间隔
- 增强去重规则
- 要求用户在区域内多做视角变化

## 4.5 R5：类别失衡严重

### 现象

- 高频类别准确率高
- 低频类别几乎学不会
- 总准确率看起来还行，但实际不可用

### 常见原因

- 用户在某些区域停留太久
- 某些类别采样不足

### 排查方法

1. 看 `class_balance.json`
2. 看每类准确率
3. 看失败样本是否集中在低频类别

### 应对策略

- 补采低频类别
- 控制采集配额
- 训练时做类别权重或采样平衡

## 4.6 R6：坏帧太多

### 现象

- 数据集里有黑屏、模糊、切出窗口画面
- 在线效果比离线差很多

### 常见原因

- 切窗口
- 快速甩视角
- 截图时机不稳定

### 排查方法

1. 抽查样本图
2. 看坏帧过滤比例
3. 看失败样本是否大量是模糊图

### 应对策略

- 增加坏帧过滤
- 增加窗口焦点校验
- 采集时减少极端动作

## 4.7 R7：字典不一致

### 现象

- `place_token` 和 `place_id` 对不上
- train 和 infer 使用的类别顺序不一致
- 标签正确但预测显示错位

### 常见原因

- 多份字典并存
- 手工修改 JSON 后未同步

### 排查方法

1. 检查 `place_dictionary.json`
2. 检查 train / infer 是否使用同一字典版本
3. 检查样本清单中的 token / id 对应关系

### 应对策略

- 全流程只允许一份正式字典
- 启动时打印字典版本
- 构建阶段做一致性校验

## 4.8 R8：训练集与测试集泄漏

### 现象

- 离线指标很好
- 在线表现却很差

### 常见原因

- 同一连续段样本同时落入 train 和 test
- 同一站位连续帧被拆散到不同 split

### 排查方法

1. 检查 split 规则
2. 看 session 或 segment 是否跨 split
3. 检查样本来源分布

### 应对策略

- 按 session 或 segment 做切分
- 避免按单帧随机切分

## 4.9 R9：在线推理不稳定

### 现象

- 实时输出不断抖动
- 同一地点预测频繁跳类

### 常见原因

- 单帧本身不稳定
- 在线截图质量差
- 区域边界本来就模糊

### 排查方法

1. 先比对离线图和在线截图的差异
2. 检查是否是某些区域特定抖动
3. 观察是否只在边界位置抖动

### 应对策略

- 先增加结果平滑
- 若仍不够，再考虑进 Version 2 引入时序能力
- 不要在 Version 1 里过早把问题升级成复杂闭环问题

## 4.10 R10：过早引入复杂结构

### 现象

- 问题一多就想加更多网络
- 训练、数据、工具问题混在一起
- 无法判断真正失败点

### 常见原因

- 还没把单帧分类测清楚就引入 RNN / 多分支 / 联合训练

### 排查方法

1. 看当前问题是否已被最小方案验证
2. 问自己：是不是基础数据链路还没稳定

### 应对策略

- 先坚持最小方案
- 先把 `cnn + mlp` 跑通
- 基础稳定后再扩结构

## 5. 推荐的排查顺序

Version 1 出问题时，建议固定按下面顺序排查：

### 排查顺序 1

先看采集条件是否一致。

### 排查顺序 2

再看标签是否切换正确。

### 排查顺序 3

再看数据集是否失衡、泄漏、坏帧过多。

### 排查顺序 4

再看训练和推理是否使用同一字典与同一类别顺序。

### 排查顺序 5

最后才看网络结构是否需要调整。

## 6. 什么时候应该补数据

建议补数据的信号：

- 某些类别样本明显太少
- 某类准确率极低但标签定义合理
- 失败样本主要来自覆盖不足

## 7. 什么时候应该改标签设计

建议改标签设计的信号：

- 两类长期强混淆
- 即使人工也不容易稳定区分
- 细粒度区域远超 Version 1 目标

## 8. 什么时候才考虑改网络

只有在以下前提基本成立后，才建议改网络：

1. 采集条件稳定
2. 标签可信
3. 数据集切分合理
4. 类别失衡已控制
5. 坏帧过滤已做

否则，优先不要把问题推给网络结构。

## 9. 最重要的结论

Version 1 最大的风险，不是模型不够复杂，而是：

> **基础数据链路不稳定。**

所以真正的优先级应始终是：

1. 先稳采集
2. 再稳标签
3. 再稳数据集
4. 最后才稳模型
