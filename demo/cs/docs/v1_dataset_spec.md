# Version 1 数据集生成与标注规范

## 1. 文档目标

本文档只服务于：

> **Version 1：区域识别**

目标是回答三个问题：

1. 训练数据如何生成
2. 用户需要做什么
3. 哪些工作必须由工具自动化完成

## 2. 基本判断

用户无法手工逐张提供足够量的数据。  
因此 Version 1 必须采用：

> **“用户做简单行为 + 工具自动采集 + 工具自动整理”**

用户可接受的最小人工工作建议为：

- 打开 CS1.6
- 进入固定设置
- 进入 `de_dust2`
- 按要求在地图里随意走动、转动视角
- 在少数时刻告诉系统“我当前在哪个区域”

除此之外，数据采样、切帧、去重、切分、打包，应尽量自动化。

## 3. Version 1 的数据来源策略

不建议一开始追求“完全自动标注”。  
Version 1 最稳妥的策略是：

### 3.1 用户负责的极少量动作

- 按规定进入固定地图
- 在区域内随意走动
- 切换到新区域时，按一次热键或输入一次区域 token

### 3.2 工具负责的自动化工作

- 自动检测并附着游戏窗口
- 自动截图
- 自动记录时间戳
- 自动记录当前激活的区域标签
- 自动保存 session 元数据
- 自动做抽帧与采样节流
- 自动去除重复帧
- 自动构建训练/验证/测试集

## 4. 推荐的数据采集模式

Version 1 建议只用下面两种模式。

## 4.1 模式 A：区域驻留采集

这是最推荐的主模式。

流程：

1. 用户进入某个区域，例如 `mid`
2. 用户通知工具当前区域 token
3. 用户在该区域内自由移动、左右看、前后走
4. 工具持续截图并自动打上该区域标签
5. 用户移动到下一个区域，再切换 token

优点：

- 标注成本很低
- 标签稳定
- 适合区域识别

## 4.2 模式 B：区域切换采集

用于补充边界样本。

流程：

1. 用户从一个区域走到另一个区域
2. 进入新区域后切换当前 token
3. 工具继续采集

这能补充：

- 区域边界视角
- 过渡视角
- 更接近真实运行的帧

## 5. 推荐的软件与工具组成

Version 1 为了完成数据生成，建议具备以下工具。

## 5.1 必需软件

### 用户侧软件

- CS1.6
- 本仓库中的 `demo/cs` 数据采集工具

### 项目侧工具

建议未来在 `demo/cs/tools/` 下提供：

1. `cs_capture_session`
2. `cs_label_session`
3. `cs_dataset_build`
4. `cs_dataset_report`

## 5.2 工具职责

### `cs_capture_session`

职责：

- 附着游戏窗口
- 固定频率截图
- 记录 session 信息
- 保存原始帧

### `cs_label_session`

职责：

- 接收用户当前区域 token
- 把该 token 绑定到一个采集时段
- 输出时间段标签文件

### `cs_dataset_build`

职责：

- 读取原始帧和时段标签
- 自动生成样本清单
- 去重
- 节流
- 切分 train / val / test

### `cs_dataset_report`

职责：

- 输出每类样本数量
- 输出每个 session 覆盖情况
- 输出重复率和采样统计

## 7. 原始采集记录格式建议

## 7.1 session.json

建议记录：

- `session_id`
- `map_name`
- `resolution`
- `capture_fps`
- `start_time`
- `end_time`
- `player_team`
- `notes`

## 7.2 label_segments.json

建议按时间段记录标签：

- `start_frame`
- `end_frame`
- `place_token`
- `place_id`

这样用户不需要逐帧标注。

## 8. 自动化处理规则

## 8.1 采样节流

不建议保存每一帧都作为训练样本。  
建议工具自动做采样节流，例如：

- 原始采集可以较高频
- 训练样本按固定间隔抽样

## 8.2 去重

建议工具自动去除：

- 完全重复帧
- 几乎无差别帧
- 极低信息量帧

这样能减少：

- 数据量膨胀
- 无效重复训练

## 8.3 模糊帧过滤

建议工具自动过滤：

- 严重模糊帧
- 黑屏帧
- 非游戏画面帧

## 8.4 类别均衡

建议工具自动统计各区域样本量。  
若发现某些区域明显偏少，工具应给出提示，而不是让用户自己手工统计。

## 9. 用户实际操作流程

下面给出一个 Version 1 可执行的最小数据采集流程。

## 9.1 准备

用户做：

1. 打开 CS1.6
2. 设置固定分辨率与固定画面参数
3. 进入 `de_dust2`

工具做：

1. 附着游戏窗口
2. 校验窗口尺寸
3. 开始截图准备

## 9.2 开始采集

用户做：

1. 进入 `t_spawn`
2. 执行“开始采集”
3. 告诉工具当前 token 是 `t_spawn`

工具做：

1. 开始保存截图
2. 开始记录当前时段标签

## 9.3 区域内自由移动

用户只需要：

- 在该区域内随意走动
- 左右转头
- 前后轻微移动

工具自动：

- 连续截图
- 自动记录当前时段属于 `t_spawn`

## 9.4 切换到下一个区域

用户做：

1. 移动到 `mid`
2. 告诉工具当前 token 切换为 `mid`

工具做：

1. 结束上一段标签
2. 开启新标签时段

## 9.5 结束采集

工具自动：

- 保存 session 元数据
- 生成标签分段文件
- 生成初步统计

## 10. 推荐的最小采集任务

Version 1 不需要一次采全图。  
建议先做最小闭环：

- `t_spawn`
- `ct_spawn`
- `mid`
- `a_site`
- `b_site`

等这 5 类稳定后，再扩到：

- `long_doors`
- `catwalk`
- `upper_tuns`

## 11. 为什么不建议 Version 1 追求完全自动标注

因为 Version 1 的目标是：

- 快速建立稳定数据链路
- 快速验证区域识别可不可行

如果一开始就追求：

- 自动知道玩家在哪
- 自动完成标签

那实际上是在先做：

- 定位
- 规则引擎
- 甚至另一个更复杂系统

这会把 Version 1 复杂度拉高。

所以更合理的原则是：

> **让用户只做极少量“段级标注”，其余全部自动化。**

## 12. Version 1 数据生成的完成定义

只有满足以下条件，才算数据链路完成：

1. 用户能按固定流程启动采集
2. 用户只需少量输入 token
3. 工具能自动生成原始 session 数据
4. 工具能自动生成处理后的训练/验证/测试清单
5. 工具能自动输出数据分布报告
6. 数据能直接供 Version 1 train 程序使用
