# CS Demo 模块版本路线图

## 1. 文档目标

把 `demo/cs` 的开发路线改成严格串行的单模块版本制。

基本原则：

- 一个版本只做一个功能
- 当前版本必须先开发完成
- 当前版本必须先测试完成
- 当前版本结论必须记录
- 然后才能进入下一个版本

## 2. 总体原则

### 2.1 版本边界

每个版本必须明确：

- 输入是什么
- 输出是什么
- 使用哪些网络
- 不做什么
- 如何测试
- 通过标准是什么

### 2.2 禁止事项

禁止：

- 在同一个版本里同时做区域识别、朝向、路线规划、动作控制
- 当前版本未验收就启动下一个版本开发
- 用“先全做出来再一起测”的方式推进

## 3. 推荐版本拆分

## Version 1：区域识别

目标：

- 输入截图
- 输出当前区域标签

推荐网络：

- `cnn + mlp`

可选增强：

- `cnn + rnn + mlp`

不做：

- 朝向识别
- 路线规划
- 动作控制

测试重点：

- 分类准确率
- 混淆区域分析
- 推理稳定性

完成标准：

- 区域标签可稳定输出
- 有测试集结果
- 有失败样本记录

## Version 2：朝向与局部运动状态

前置条件：

- Version 1 已通过

目标：

- 输入连续帧
- 输出朝向桶与局部运动状态

推荐网络：

- `cnn + rnn + mlp`

不做：

- 路线规划
- 动作控制

测试重点：

- 朝向分类准确率
- 运动状态识别准确率
- 连续帧稳定性

## Version 3：路线规划

前置条件：

- Version 2 已通过

目标：

- 输入当前区域 ID 与目标区域 ID
- 输出下一跳区域或路径提示

推荐网络：

- `gnn + mlp`

说明：

- 这一版可以先不接截图
- 先把地点图规划单独做好

测试重点：

- 路径正确率
- 下一跳正确率
- 图结构边界条件

## Version 4：目标编码与控制输入准备

前置条件：

- Version 3 已通过

目标：

- 把 `place_token` 稳定转换成控制侧目标表示
- 打通目标输入、当前区域、路线提示的联合输入格式

推荐网络：

- `mlp`

说明：

- 这一版重点是数据接口，不是复杂动作控制

测试重点：

- 输入字典一致性
- `place_token -> place_id` 一致性
- 联合输入张量格式正确性

## Version 5：动作策略影子模式

前置条件：

- Version 4 已通过

目标：

- 输入感知与规划结果
- 输出建议动作
- 只打印动作，不发送按键

推荐网络：

- `mlp`
- 或 `rnn + mlp`

测试重点：

- 动作分类稳定性
- 典型场景输出是否合理
- 错误动作样本分析

## Version 6：闭环移动控制

前置条件：

- Version 5 已通过

目标：

- 接入真实按键执行
- 完成闭环移动
- 完成急停、超时、失败回退

测试重点：

- 卡键风险
- 焦点丢失停机
- 目标到达判定
- 闭环稳定性

## 4. 推荐执行顺序

严格顺序如下：

1. Version 1：区域识别
2. Version 2：朝向与局部运动状态
3. Version 3：路线规划
4. Version 4：目标编码与控制输入准备
5. Version 5：动作策略影子模式
6. Version 6：闭环移动控制

## 5. 当前最推荐先做的版本

当前最推荐：

> **Version 1：区域识别**

原因：

- 最容易独立定义
- 最容易独立测试
- 最容易建立数据规范
- 是后续功能的基础

## 6. 每个版本的交付物建议

每个版本都建议至少交付：

- 版本说明文档
- 数据说明文档
- generate/train/infer 入口
- 测试结果记录
- 已知问题清单

## 7. 版本切换门禁

进入下一版本前，至少满足：

1. 当前版本功能完成
2. 当前版本测试完成
3. 当前版本结果已记录
4. 当前版本遗留风险已知
5. 已确认下一个版本的输入输出边界
