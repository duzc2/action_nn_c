# Version 1 测试计划

## 1. 文档目标

本文档定义 Version 1 的测试方式。

Version 1 只测：

> **区域识别**

不测：

- 朝向识别
- 路线规划
- 动作控制
- 闭环导航

## 2. 测试目标

Version 1 的测试目标不是“看起来像能用”，而是明确回答：

1. 数据链路是否稳定
2. 训练链路是否稳定
3. 推理链路是否稳定
4. 区域识别是否达到可接受水平
5. 是否可以进入 Version 2

## 3. 测试范围

Version 1 测试分为 4 类：

1. 工具链测试
2. 数据集测试
3. 模型测试
4. 在线验证测试

## 4. 工具链测试

## 4.1 `cs_capture_session`

测试点：

- 能否正常启动
- 能否附着游戏窗口
- 能否持续截图
- 能否正常停止
- 是否生成 `session.json`
- 是否生成 `frames/`

通过标准：

- 至少完成 1 个可用 session
- 无崩溃
- 文件结构正确

## 4.2 `cs_label_session`

测试点：

- `set` 是否能切换 token
- 是否正确关闭上一标签段
- 是否正确写出 `label_segments.json`

通过标准：

- 标签段连续
- 无非法重叠
- token / id 一致

## 4.3 `cs_dataset_build`

测试点：

- 是否能读取 raw session
- 是否能读取标签文件
- 是否能生成 train / val / test
- 是否能处理去重与过滤

通过标准：

- 输出清单存在
- 样本字段完整
- 样本路径有效

## 4.4 `cs_dataset_report`

测试点：

- 是否输出每类数量
- 是否输出 split 分布
- 是否提示缺类或失衡

通过标准：

- 报告可读
- 统计值与数据清单一致

## 5. 数据集测试

## 5.1 字典一致性测试

检查：

- `place_token` 是否都存在于字典
- `place_id` 是否都与字典一致

通过标准：

- 0 个非法 token
- 0 个非法 id

## 5.2 标签段合法性测试

检查：

- 段区间是否有重叠
- 段区间是否有倒置
- `start_frame <= end_frame`

通过标准：

- 0 个非法段

## 5.3 样本路径测试

检查：

- `train_list.json`
- `val_list.json`
- `test_list.json`

中的图片路径是否都存在。

通过标准：

- 0 个丢失样本路径

## 5.4 类别覆盖测试

检查：

- 每个启用类别是否都有样本
- train / val / test 是否至少覆盖核心类别

通过标准：

- 核心类别不缺失

## 5.5 类别均衡测试

检查：

- 是否存在某类样本极少
- 是否存在单类严重偏大

通过标准：

- 无明显极端失衡

注：

Version 1 不要求绝对平均，但必须可训练。

## 6. 模型测试

## 6.1 训练可运行性测试

检查：

- train 程序是否可读数据集
- 是否能完成至少一次完整训练
- 是否能输出权重文件

通过标准：

- 训练流程跑通
- 权重产出成功

## 6.2 推理可运行性测试

检查：

- infer 程序是否能读取图片
- 是否能输出 `place_id`
- 是否能输出 `place_token`
- 是否能输出置信度

通过标准：

- 单图推理成功
- 输出格式稳定

## 6.3 离线准确率测试

必须统计：

- 总准确率
- 每类准确率
- 混淆矩阵

建议验收线：

- 总准确率先达到一个“可继续开发”的水平
- 每类不能全靠个别类别拉高平均值

建议初始标准：

- 总准确率达到 **70%+**
- 核心 5 类中大多数类别达到可用水平

这只是建议线。  
真正阈值后续可以再收紧。

## 6.4 失败样本测试

必须保留：

- 真实标签
- 预测标签
- 置信度
- 原始图片路径

通过标准：

- 能抽取失败样本
- 能复盘错误来源

## 7. 在线验证测试

## 7.1 实时截图推理测试

目标：

- 在真实运行中的 `de_dust2` 里实时输出区域预测

检查：

- 是否能稳定获取截图
- 是否能持续输出预测结果
- 是否能正常退出

通过标准：

- 连续运行一段时间无崩溃
- 输出可观察

## 7.2 人工观察测试

做法：

- 用户在已知区域停留
- 观察输出区域是否大体正确

通过标准：

- 多个典型区域预测基本符合预期

## 8. 测试记录要求

Version 1 必须保留测试记录。

建议至少记录：

- 测试日期
- 使用的数据集版本
- 使用的字典版本
- 使用的模型配置
- 总准确率
- 每类准确率
- 失败样本列表
- 是否通过

## 9. 建议的测试产物

建议生成：

- `evaluation_summary.md`
- `confusion_matrix.json`
- `failure_cases.json`
- `online_validation_notes.md`

## 10. Version 1 的验收结论类型

测试结束后，建议只允许给出以下 3 类结论之一：

### 结论 A：通过

说明：

- 数据链路稳定
- 训练链路稳定
- 推理链路稳定
- 区域识别达到可接受水平

动作：

- 可以进入 Version 2

### 结论 B：补数据后重测

说明：

- 流程没问题
- 但数据覆盖不足或标签质量不足

动作：

- 回到数据采集阶段补采

### 结论 C：方案需调整

说明：

- 不是单纯数据少
- 而是标签定义、采样策略或网络结构存在明显问题

动作：

- 先修方案，再继续

## 11. 当前最重要的测试原则

Version 1 最重要的不是“追求最好分数”，而是：

> **先判断区域识别这条路是不是稳定可走。**

所以测试必须服务于“能否继续推进”，而不是只服务于“好不好看”。
