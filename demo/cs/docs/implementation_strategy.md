# CS Demo 实现思路文档

## 1. 文档目的

本文档面向后续开发阶段，说明 `demo/cs` 的推荐实现路线，包括：

- 如何结合当前仓库已有技术框架
- 训练过程怎么组织
- 推理闭环怎么搭建
- 网络结构怎么设计
- 如果现有网络不够，应该补哪些新结构

当前阶段仍然只是方案设计，**不包含代码实现**。

## 2. 先看当前仓库可复用能力

结合当前仓库现状，已经具备以下可直接复用的网络类型：

- `cnn`
- `mlp`
- `rnn`
- `gnn`
- `transformer`

这些类型都已经符合当前工程主流程：

- 通过 CMake 开关启用
- 通过注册单元接入
- 通过 profiler 参与生成 / 训练 / 推理

因此，`demo/cs` 第一版**不必一上来新增很多底层类型**，完全可以先基于：

> `cnn + rnn + gnn + mlp`

做出一个强度已经比较高的演示。

## 3. 推荐的总体实现路线

新的实施原则是：

> **一个版本只实现一个功能模块。**

不再把区域识别、朝向判断、路线规划、动作控制打包成同一个“第一版”。

建议把 `demo/cs` 拆成四个能力面：

### 3.1 感知面

输入：

- 游戏窗口截图
- 连续多帧图像

输出：

- 当前 callout 区域估计
- 当前朝向估计
- 局部运动状态估计

### 3.2 规划面

输入：

- 当前区域
- 目标区域
- `de_dust2` 的地点邻接图

输出：

- 下一段应该前往的子目标
- 大方向引导

### 3.3 控制面

输入：

- 感知特征
- 规划特征
- 当前目标

输出：

- 当前一步动作

### 3.4 执行面

输入：

- 动作决策

输出：

- 键盘输入
- 日志
- 急停与回退

## 4. 推荐的分层网络结构

我建议优先考虑下面这套结构。

```text
agent
├─ perception_group
│  ├─ frame_encoder        (cnn)
│  ├─ temporal_encoder     (rnn)
│  └─ area_head            (mlp)
├─ planning_group
│  ├─ callout_graph        (gnn)
│  └─ route_head           (mlp)
└─ control_group
   ├─ goal_encoder         (mlp)
   └─ action_head          (mlp 或 rnn)
```

## 5. 各子网职责

### 5.1 frame_encoder（推荐：`cnn`）

作用：

- 对截图做压缩编码
- 提取局部纹理、结构、亮暗、边缘等视觉特征

建议输入：

- 单帧灰度图
- 或连续 2~4 帧的小尺寸图像

建议输出：

- 一个中小维度视觉特征向量

### 5.2 temporal_encoder（推荐：`rnn`）

作用：

- 把连续帧特征融合成时序状态
- 稳定当前位置与朝向判断
- 降低单帧误判影响

建议输入：

- 最近 N 帧的 `frame_encoder` 输出

建议输出：

- 稳定的时序隐藏状态

### 5.3 area_head（推荐：`mlp`）

作用：

- 根据时序特征预测当前所处 callout
- 可同时预测朝向桶（heading bucket）

建议输出：

- 当前区域分类
- 当前朝向分类

### 5.4 callout_graph（推荐：`gnn`）

作用：

- 利用 `de_dust2` 的地点图做高层导航
- 输入当前区域和目标区域，输出推荐下一跳或下一段路线

这是非常适合展示当前项目组织能力的一层，因为：

- 图结构天然适合表示 callout 邻接关系
- `de_dust2` 本身就能抽象成一个固定地点图
- 可以把“唯一地点标识符”直接转成图节点

建议节点表示：

- `is_current`
- `is_goal`
- `is_active`
- `x_norm`
- `y_norm`
- `region_type`

建议输出：

- 下一跳 callout
- 或者一组方向偏好特征

### 5.5 route_head（推荐：`mlp`）

作用：

- 把 `gnn` 输出的图级 / 锚点级表示转成可供控制器使用的路线提示

### 5.6 goal_encoder（推荐：`mlp`）

作用：

- 把目标地点整数 ID、下一跳地点 ID、当前区域 ID、相对误差编码成控制特征

### 5.7 action_head（推荐：`mlp`；增强版可用 `rnn`）

作用：

- 输出一步动作

建议动作空间先做成离散分类：

- `forward`
- `backward`
- `strafe_left`
- `strafe_right`
- `turn_left_small`
- `turn_right_small`
- `stop`

不建议第一版直接回归连续控制量。

## 6. 推荐的“规则 + 网络”分工

第一版最稳妥的方式不是全神经端到端，而是混合架构。

### 6.1 规则模块负责

- 窗口查找
- 焦点校验
- 急停
- 非法状态拦截
- 目标标识符合法性检查
- 目标标识符到整数 ID 的转换
- `de_dust2` 地点图构建
- 是否到达目标区域的判定
- 动作持续时长裁剪

### 6.2 网络模块负责

- 视觉特征提取
- 当前区域识别
- 时序状态压缩
- 路线提示推断
- 当前一步动作决策

这更符合仓库现有原则：

- 用流程保证正确性
- 用网络解决复杂模式识别

## 7. 训练过程建议

我建议把训练拆成**多个独立版本**，不要一开始就把所有任务混在一起。

### 7.1 阶段 A：区域识别预训练

目标：

- 先让模型学会“我现在在哪”

训练数据：

- `de_dust2` 不同地点的截图
- 每张图对应一个 callout 标签

输入：

- 单帧或短帧序列

输出：

- callout 分类

推荐结构：

- `cnn -> mlp`
- 或 `cnn -> rnn -> mlp`

验收标准：

- 在固定分辨率、固定画面设置下，常用区域能稳定区分

### 7.2 阶段 B：朝向与局部运动状态训练

目标：

- 学会判断自己面向哪里、在移动还是卡住

训练数据：

- 连续截图
- 对应朝向桶标签
- 可选的移动状态标签

推荐结构：

- `cnn -> rnn -> mlp`

输出：

- heading bucket
- progress state

### 7.3 阶段 C：路线规划训练

目标：

- 在地点图上学习从当前区域到目标区域的高层路径提示

训练数据：

- `de_dust2` callout 图
- 当前节点
- 目标节点
- 教师路径下一跳

推荐结构：

- `gnn -> mlp`

教师标签来源：

- 手工定义邻接图
- BFS / 最短路径算法得到的下一跳

这一阶段很重要，因为它把：

- 地点标识符
- 地图结构
- 高层导航

三者连接起来了。

### 7.4 阶段 D：动作策略训练

目标：

- 根据视觉状态 + 高层路径提示，学会输出一步动作

训练数据：

- 连续截图
- 当前区域
- 目标区域
- 下一跳区域
- 教师动作

推荐结构：

- `(cnn + rnn) + (gnn + mlp) + mlp`
- 或者最后一层 `action_head` 用 `rnn`

教师动作来源建议：

- 人工示教
- 规则控制器
- waypoint 教师器

第一版最推荐：

> **规则控制器先产出可用 teacher action，网络先做 imitation learning。**

这样更容易收敛。

## 8. 数据采集建议

### 8.1 数据采集环境

建议固定：

- 地图：`de_dust2`
- 分辨率
- 亮度 / 贴图设置
- FOV / HUD 风格
- 本地 bot 房或空房

### 8.2 数据记录内容

建议每条记录至少包括：

- 时间戳
- 原始截图路径
- 当前 callout 标签
- 目标 callout ID
- 朝向标签
- 教师动作标签
- 上一动作
- 是否到达目标

建议额外固定一份字典文件：

- `place_token -> place_id`

这样训练数据、运行参数、网络输入三者始终一致。

### 8.3 数据来源建议

推荐顺序：

1. 手工采集静态截图，先做区域识别
2. 手工跑简单路线，录连续帧
3. 规则控制器生成 teacher action
4. 少量人工复核

### 8.4 数据切分

建议按以下维度切分训练/验证：

- 按地点切分
- 按路线切分
- 按出生点切分

避免同一路线序列同时出现在训练和验证中。

## 9. generate / train / infer 三阶段怎么落

每一个版本都应单独遵守 generate / train / infer 主流程；  
不要把多个模块合并到同一版里再统一生成和验证。

## 9.1 generate 阶段

职责：

- 定义复合网络拓扑
- 声明 `cnn`、`rnn`、`gnn`、`mlp` 的组合关系
- 为训练与推理分别输出生成代码

这里应保持与仓库主流程一致：

- 用户侧构建网络结构体
- 调用 profiler
- 输出训练/推理 `.c` 与固定 `.h`

## 9.2 train 阶段

职责：

- 读取离线数据集
- 先做子任务训练，后做联合训练
- 产出权重 `.bin` 与权重 `.c`

推荐训练组织方式：

1. 先训练 `perception_group`
2. 再训练 `planning_group`
3. 最后训练 `control_group`
4. 最后可加少量端到端微调

## 9.3 infer 阶段

职责：

- 启动或附着 CS1.6
- 截图
- 校验目标地点标识符
- 把目标地点标识符转换为整数 ID
- 加载权重
- 执行闭环控制

## 10. 第一版最推荐的最小可行方案

在新的串行版本策略下，这一节应理解为：

> **先做最小单模块版本，而不是先做完整导航版。**

如果目标是“尽快做出强 demo”，最推荐的不是最大而全，而是：

### 10.1 Phase 1 推荐方案

- 地图固定为 `de_dust2`
- 只支持 10~20 个常用 callout
- 只支持固定分辨率
- 只支持本地 bot 房
- 只做移动，不做开火
- 只做几个典型路线：
  - `T Spawn -> Mid`
  - `T Spawn -> Long Doors`
  - `T Spawn -> B Site`
  - `CT Spawn -> A Site`

### 10.2 推荐网络

- 感知：`cnn + rnn + mlp`
- 规划：`gnn + mlp`
- 控制：`mlp`

这套已经足够展示：

- 视觉
- 图结构规划
- 时序建模
- 组合网络
- 主链路生成训练推理

## 11. 如果现有网络还不够，建议新增哪些结构

当前并不建议为了第一版一下子新增很多后端。  
但如果在实现过程中发现现有结构不够，下面这些新增类型是合理的。

### 11.1 `siamese_metric`（优先级高）

定位：

- 做地点相似度匹配
- 适合“当前画面更像哪个 callout”的度量学习

适用场景：

- 某些区域视觉差异细微
- 纯分类头不稳定

接入方式建议：

- 按 `src/nn/types/siamese_metric/` 新增
- 补 config-only 头文件
- 补训练 / 推理 ops
- 通过注册单元接入

### 11.2 `temporal_conv` 或 `tiny_tcn`（优先级中）

定位：

- 比简单 RNN 更稳定地建模短时序

适用场景：

- 需要用最近几帧判断转向趋势
- RNN 容易抖动

### 11.3 `spatial_heatmap_head`（优先级中）

定位：

- 输出一个地点热力图或局部方位热图

适用场景：

- 只做分类不够，需要更细粒度位置感

### 11.4 `policy_head`（优先级低）

定位：

- 专门承载离散动作策略输出

说明：

- 第一版其实用 `mlp` 就够
- 除非后面动作空间明显扩大，否则不建议单独拆新类型

## 12. 推荐的开发顺序

当前更推荐把这里理解为“版本顺序”而不是“一次性大工程顺序”。

### Step 1

先把 `de_dust2` 的地点标识符、地点 ID、邻接图、目标区域定义清楚。

### Step 2

先做离线截图采集和 callout 分类验证。

### Step 3

再做 `gnn` 路径提示分支。

### Step 4

再做动作 teacher 与 imitation learning。

### Step 5

最后接实时截图和键盘闭环。

## 13. 关键风险与应对

### 风险 A：视觉识别不稳定

应对：

- 固定地图
- 固定分辨率
- 先做 callout 分类
- 引入短时序融合

### 风险 B：目标地点太细，难以稳定识别

应对：

- 第一版只支持大区域和高辨识度点位
- 不追求过细 callout

### 风险 C：高层路径和低层动作互相打架

应对：

- 规划输出只做“下一跳/方向提示”
- 控制层保留局部修正权

### 风险 D：闭环控制抖动

应对：

- 动作做离散分类
- 增加短时保持与最小时长
- 用规则做节流和急停

## 14. 最终建议

我对这个 demo 的建议是：

> **先把 Version 1 的区域识别单独做完、单独测通。**

原因：

- 最符合仓库现有主链路
- 能充分展示“已有网络类型 + 复杂组织能力”
- 不需要一开始就扩太多底层后端
- 后续如果视觉定位不够稳，再补 `siamese_metric` 这类新网络类型

换句话说：

- **第一版重点是把 demo 做成**
- **第二版再考虑把网络做得更花**
